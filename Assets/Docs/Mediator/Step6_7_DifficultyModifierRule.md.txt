# Step 6.7 – DifficultyModifier Rule (난이도 조정 규칙 고정)

## 0) 범위
- 본 문서는 Step 6.7(구조/문서 설계 단계)만 다룬다.
- 실제 수치 밸런싱, UI 구현, Step 7 이상 구현은 포함하지 않는다.

## 1) 공통 규칙(절대 고정)
- DifficultyModifier는 판단자가 아니라 **룰 기반 조정 데이터**다.
- 최종 판단/조합/적용은 **Mediator**에서만 수행한다.
- Event Layer는 전달/기록만 수행한다.
- Executor는 Command 실행만 수행한다.
- 타입 분기 금지:
  - `if (difficultyId == "...")`
  - `switch (difficultyId)`
  - 난이도 이름 기반 하드코딩 분기 전부 금지

## 2) Step 6.7 목표
- “대중성 기본 + 하드코어 옵션”을 구조로 고정한다.
- 난이도가 조정 가능한 축(속도/패턴 추가/제약/보상)을 데이터 노브로 정의한다.
- Boss(6.5), Promotion(6), PlayerTalent(6.6)와 충돌하지 않게 경계를 고정한다.

## 3) 난이도 모드 최소 구조
- Easy/Normal/Hard는 표시 이름일 뿐이며, 실체는 `difficultyId + ruleSet`이다.
- 난이도 선택 결과는 활성 `difficultyId`로 관리한다.
- 난이도는 스스로 판단하지 않는다(데이터 제공만).

## 4) 데이터 모델 최소 필드 (SO or Data)
## 4.1 DifficultyDefinition
- `difficultyId: string` (예: `difficulty.normal`)
- `tags: string[]` (메타데이터용)
- `rules: DifficultyRule[]`
- `modifiers: Modifier[]`
- `constraints: string[]`
- `rewardModifiers: Modifier[]` (선택)

## 4.2 DifficultyRule
- `ruleKey: string`
- `targetKey: string`
- `op: string` (`enable` / `disable` / `set_flag`)
- `params: key/value` (선택)
- 비고: Step6.7 기준에서 `rules`는 **구조/활성화/제약 토글**만 담당한다.

## 4.3 Modifier
- `statKey: string`
- `value: float`
- 해석 규약:
  - `stat.xxx.mul` -> 배수값 (예: `1.10`)
  - `stat.xxx.add` -> 델타값 (예: `+5`)
- 비고: Step6.7 기준에서 `modifiers`는 **순수 수치 보정**만 담당한다.

## 5) rules vs modifiers 역할 분리 (중복 해석 방지)
- `rules`는 구조/활성화/제약 토글 전용.
- `modifiers`는 수치 보정 전용.
- 동일 효과를 `rules` + `modifiers`로 이중 적용하지 않는다.

## 6) targetKey 허용 범위(강제 제한)
- 허용 prefix:
  - `system.tempo.*`
  - `pattern.*`
  - `constraint.*`
  - `reward.*`
- 금지:
  - `boss.*`, `weapon.*`, `promotion.*`, `talent.*` 내부 필드 직접 타겟팅
- 정책:
  - 등록된 Key 카탈로그에 존재하는 `targetKey`만 유효
  - 미등록 key는 무시/거부 + 사실 이벤트 기록

## 7) 난이도 조정 축(허용 목록)
- 전투 템포/속도 계수
- 추가 패턴 키 활성화/비활성화
- 제약 키 활성화(예: 힐 제한)
- 보상 계수 조정

## 8) Boss/Promotion/Talent 경계선
- Difficulty는 패턴 후보군/가중치/활성 플래그/제약/보상만 조정한다.
- **페이즈 전환 조건(HP/시간/기믹 완료)은 Boss 규칙 영역**으로 유지한다.
- Promotion/Talent의 정의/선택 규칙을 Difficulty가 대체하지 않는다.

## 9) 단일 합성 지점 원칙
- 최종 합성은 **단일 Compose Mediator 역할**에서만 수행한다.
- 이름은 구현 단계에서 결정(예: `BattleModifierComposeMediator` 역할).
- 개별 Mediator는 입력 수집/검증까지만 담당한다.

## 10) 조합 우선순위(고정)
1. Base
2. Promotion
3. PlayerTalent
4. Difficulty
5. Clamp

## 11) Clamp 규약
- Clamp는 최종 수치 범위(min/max) 보정만 담당한다.
- 기능 토글(enable/disable), 후보군 활성화는 Clamp 대상이 아니다.

## 12) tags 사용 규약
- `tags`는 UI/필터/표시용 메타데이터로만 사용한다.
- 판단 로직 분기에 사용하지 않는다.
- 처리 로직은 `rules/modifiers` 순회 기반으로만 수행한다.

## 13) 전투 중 난이도 변경 기본 정책
- Step6.7 기본 정책: **전투 중 난이도 변경 거부**.
- 난이도는 로비/시작 전 구간에서 확정한다.
- 전투 중 변경 요청은 사실 이벤트로 기록한다:
  - `DifficultyChangeRejectedOccurred(reason)` 또는 `RequestIgnoredOccurred(reason)`

## 14) Step 7 진입 불변 조건
- Difficulty는 데이터 제공자이며 판단자가 아니다.
- 판단/합성/적용은 Mediator 단일 지점에서만 수행한다.
- 타입 분기 금지 원칙을 유지한다.
- Boss/Promotion/Talent/Weapon 경계를 침범하지 않는다.
- 우선순위 `Base -> Promotion -> Talent -> Difficulty -> Clamp`를 유지한다.
