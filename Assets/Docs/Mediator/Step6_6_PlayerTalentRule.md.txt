# Step 6.6 – PlayerTalent System Rule (구조/문서 설계 고정)

## 0) 범위
- 본 문서는 Step 6.6(문서 설계 단계)만 다룬다.
- 구현(UI/코드/밸런스 수치 확정/특수 스킬)은 Step 7 이후로 미룬다.

## 1) 공통 규칙(절대 고정)
- PlayerTalent는 판단/분기/실행 로직을 가지지 않는다. 데이터만 제공한다.
- 판단은 Mediator만 수행한다.
- Event Layer는 전달/기록만 수행한다.
- 기존 구조는 수정하지 않고 추가만 한다.
- Step 1~5 책임 경계를 절대 무너뜨리지 않는다.

## 2) PlayerTalent 정체성 고정
- 게임 시작 시 선택한다.
- 정확히 3개 장착한다.
- 전투 시작 후 변경 불가(고정).
- 성장형 없음(레벨업/강화로 Talent 자체가 커지지 않음).
- 효과는 미세 보정 중심(권장: 5~10%대).

## 3) PlayerTalent가 정의할 수 있는 것
- 이동/생존/숙련도 획득/리듬(템포) 관련 보정 입력
- tags 기반 분류 입력
- effectKeys + modifiers 기반 보정 입력

## 4) PlayerTalent가 절대 하지 않는 것
- 조건 체크/분기/타겟 선택
- Command 발행/실행
- 연출 호출(VFX/Animation/UI)
- 하드코딩 타입 분기
  - 금지 예: if (talentId == "..."), switch (talentId), if (weaponId == "...")

## 5) 최소 데이터 스키마(고정)
### 5.1 PlayerTalentDefinition (SO or data)
- talentId: string
- tags: string[]
- effectKeys: string[]   (선택: 빈 배열 가능)
- modifiers: Modifier[]

### 5.2 PlayerTalentLoadout (전투 시작 입력)
- actorId: string
- equippedTalentIds: string[3]   (정확히 3개, 중복 불가)

### 5.3 Modifier
- statKey: string
- value: float

## 6) effectKeys vs modifiers 역할 분리(중복 방지)
- 원칙: 단순 수치 보정은 modifiers로만 표현한다.
- effectKeys는 규칙 테이블 해석이 필요한 경우(계산 방식 선택)만 사용한다.
- Step6.6 MVP에서는 effectKeys는 선택(빈 배열 가능)으로 둔다.

## 7) value 해석 규약(혼동 방지)
- stat.xxx.mul  -> value는 배수값 (예: 1.05 = +5%)
- stat.xxx.add  -> value는 델타값 (예: +5)

## 8) Event Boundary / 거부 정책
- 전투 시작 전: Loadout 고정 요청 허용
- 전투 시작 후: 변경 요청은 허용하지 않는다.
- 전투 중 변경 요청은 사실 이벤트로 기록한다.
  - TalentChangeRejectedOccurred(reason) 또는 RequestIgnoredOccurred(reason)
  - (정책 선택/구체 이벤트명은 Step7에서 확정)

## 9) Mediator 조합 원칙(단일 판단 지점)
- Loadout 유효성(정확히 3개/중복 없음/ID 존재) 검증은 Mediator에서만 수행한다.
- 모든 Modifier 합성은 WeaponPromotionComposeMediator(또는 동일 축의 단일 Compose Mediator)에서만 수행한다.
- 개별 Mediator는 입력 수집/검증까지만 담당한다.
- 조합 순서 고정:
  1) Base
  2) Promotion
  3) PlayerTalent
  4) Difficulty
  5) Clamp

## 10) Core 결합 리스크 완화 문장(고정)
- Step7~8에서 Core는 talentId + effectKeys + modifiers(값 타입)만 의존하고,
  PlayerTalentDefinitionSO 조회는 Composition Root/Adapter에서 제공한다.

## 11) Step 7 진입 불변 조건
- Talent는 데이터 제공자이며 판단/실행을 하지 않는다.
- 판단은 Mediator 단일 지점에서만 발생한다.
- Executor는 실행만 하고 판단하지 않는다.
- 타입 분기 금지 원칙을 유지한다.
- Loadout은 전투 시작 시점에 고정되고 이후 변경되지 않는다.
